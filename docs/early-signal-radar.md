# Early Signal Radar

## 最終目標

しえすたんBotを、Nansenのオンチェーンデータを使ってSmart Moneyやアルトコイン周辺の初期シグナルを検知するDiscord Botへ拡張する。

最終的には、4時間ごとに候補トークンをスキャンし、異常な資金流入、Smart Moneyの保有変化、出来高やホルダー変化などを総合して、Discordに見やすく通知する。

このBotは投資判断を自動化するものではなく、調査のきっかけを早く見つけるためのレーダーとして扱う。

## MVP機能

- 手動コマンドでNansen CLIの接続確認を行う
- 手動コマンドでSmart Money関連データを取得する
- 取得結果をDiscord Embedで表示する
- `!deep solana TOKEN_ADDRESS` で候補トークンを追加データで深掘りする
- 対象チェーン、対象トークン、取得期間を固定値または設定値として扱う
- エラー時はDiscordには短い失敗メッセージだけを出し、詳細はログに出す
- APIクレジット消費が大きい処理は明示的なコマンド実行に限定する

## 将来の4時間定期スキャン構成

将来的には、以下の流れを4時間ごとに実行する。

1. 対象チェーンと候補トークンを決める
2. Nansen CLIでSmart Money関連データを取得する
3. 必要に応じてトークン情報、フロー、ホルダー、PnLなどを取得する
4. 取得結果を正規化してシグナル候補として保存する
5. 4-Gate設計に基づいて候補を絞り込む
6. スコアリングして通知優先度を決める
7. Discordに高スコアの候補だけを投稿する
8. 実行結果、エラー、APIクレジット消費の目安をログに残す

定期実行は、最初はNode.jsプロセス内のスケジューラで十分。運用が重くなったら、外部cron、GitHub Actions、VPSのsystemd timer、キュー付きワーカー構成を検討する。

## 4-Gate設計

### Gate 1: Smart Money Flow

Smart Moneyの流入が一定以上あるかを確認する。

- netflowがプラスか
- 直近の流入が過去平均より大きいか
- Fund、Smart Traderなど複数ラベルで偏りがないか

### Gate 2: Holder Quality

買っているウォレットの質を確認する。

- Smart Moneyラベルの数
- 既存ホルダーと新規ホルダーのバランス
- 短期売買だけのウォレットに偏っていないか

### Gate 3: Token Health

トークン自体の状態を確認する。

- 流動性が極端に薄くないか
- 出来高が自然に増えているか
- 急騰後の遅すぎる検知ではないか
- ステーブルコインやノイズになりやすい大型銘柄を除外するか

### Gate 4: Confirmation

複数の根拠が揃っているかを確認する。

- Smart Money流入と出来高増加が同時に起きているか
- ホルダー増加や転送増加も伴っているか
- 同一ウォレット群だけの循環的な動きではないか

## 保存するシグナル項目

- 検知日時
- チェーン
- トークン名
- トークンシンボル
- トークンアドレス
- Nansen URLまたは参照用URL
- Smart Money netflow
- Smart Money買い越し額
- Smart Money売り越し額
- 関連ラベル
- 関与ウォレット数
- 新規Smart Moneyホルダー数
- 出来高変化率
- ホルダー数変化率
- 価格変化率
- 流動性の目安
- Gateごとの通過結果
- 総合スコア
- 通知済みかどうか
- 最終更新日時
- エラーや注意点のメモ

## スコアリング案

初期案として100点満点で評価する。

- Smart Money netflow: 30点
- Smart Moneyウォレットの質と分散: 20点
- 出来高、ホルダー、流動性の健全性: 20点
- 価格がまだ過熱しすぎていないこと: 15点
- 複数データソースまたは複数指標での確認: 15点

スコア例:

- 80点以上: Discordに強めの注目候補として通知
- 60点以上: 通常の注目候補として通知
- 40点以上: 保存のみ
- 40点未満: ノイズとして破棄または低優先度保存

## false positiveを減らすための注意点

- 価格急騰後だけに反応しない
- 流動性が薄すぎるトークンを高評価しない
- 単一ウォレットの大口移動だけで強いシグナル扱いしない
- CEX入出金、ブリッジ、内部移動を買いシグナルと誤認しない
- ステーブルコイン、ラップドトークン、大型銘柄の通常フローを過大評価しない
- 同一エンティティ内の移動やウォッシュ的な動きを除外する
- 短時間で連続通知しないようにクールダウンを入れる
- スコアの根拠をDiscordに表示し、ブラックボックス化しない

## APIクレジット節約方針

- 4時間スキャンでは最初に軽いデータだけ取得する
- Gateを通過した候補にだけ追加の詳細取得を行う
- 手動のDeep分析では、Flow Intelligence、Token Holders、DEX Tradesを必要な時だけ取得する
- 同じトークンやウォレットの結果は一定時間キャッシュする
- Discordの手動コマンドにはレート制限を入れる
- 取得対象チェーンとトークン数を設定で制限する
- 低スコア候補には高コストな詳細分析を実行しない
- 失敗時のリトライ回数を制限する
- 実行ごとに取得コマンドと件数をログに残す

## Discord表示方針

Discord通知は、短く見やすいEmbedを基本にする。

- タイトルにチェーン、シンボル、スコアを入れる
- 説明文には検知理由を1から3行でまとめる
- フィールドにnetflow、関与ウォレット数、出来高変化率、Gate通過状況を載せる
- 詳細調査用のリンクを付ける
- 高スコアだけメンションや強調表示を検討する
- 同じトークンの通知は一定時間まとめる
- エラーや内部ログはDiscordに出しすぎない

## 投資助言ではない注意書き

しえすたんBotが表示する内容は、オンチェーンデータに基づく調査補助情報であり、投資助言ではない。

通知されたトークンの購入、売却、保有を推奨するものではない。最終的な判断は、ユーザー自身の調査と責任で行う。
